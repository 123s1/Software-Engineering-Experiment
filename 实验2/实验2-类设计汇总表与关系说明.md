# 实验2：类设计汇总表与关系说明

## 1. 文档说明

本文件用于把“教室预订系统”的实验2类图设计内容，整理成**便于写实验报告的表格形式**，并补充中文说明与类之间的关系说明。

本次设计严格依据实验1中已经完成和完善的需求分析展开，重点保证以下几点：

- 类的划分符合业务语义
- 类的职责边界清晰
- 继承、关联、组合、依赖关系使用合理
- 能够支撑 Rational Rose 7.0 的逆向生成
- 能够为后续实验3数据库设计提供基础

## 2. 设计原则总结

本次类图设计遵循以下原则：

| 设计原则 | 含义 | 在本系统中的体现 |
| --- | --- | --- |
| 单一职责原则 | 一个类只负责一类核心职责 | `User` 负责用户基本信息，`Classroom` 负责教室资源，`ReservationService` 负责预订流程 |
| 高内聚 | 同一类中的属性和行为尽量紧密相关 | 时间段相关属性与操作集中在 `ClassroomTimeSlot` 中 |
| 低耦合 | 类与类之间避免无意义依赖 | 管理流程通过服务类组织，而不是所有实体类互相直接操作 |
| 合理抽象 | 抽取公共父类，避免重复 | `Administrator` 和 `TeacherStudentUser` 统一继承 `User` |
| 慎用继承 | 只有明确 is-a 关系才使用继承 | 只保留 `User` 与两个子类的继承结构 |
| 组合优于松散堆砌 | 整体与部分关系用组合表达 | `Classroom` 与 `ClassroomTimeSlot` 是强拥有关系 |
| 关系明确 | 类之间的多重性、关系类型要清晰 | 用户与预订、教室与时间段、预订与教室/时间段之间都能清晰表达 |
| 与需求一致 | 类、属性、方法命名应能映射到需求 | 登录、修改密码、预订教室、设置可用时间段等都能映射到相应类和方法 |

## 5. 这些关系为什么这样设计

## 5.1 为什么 `User` 用继承

因为管理员和师生都属于“系统用户”，具有共同属性，所以使用继承最自然。这里是典型的 is-a 关系。

## 5.2 为什么 `Classroom` 和 `ClassroomTimeSlot` 用组合

时间段不是孤立存在的，它必须依附于具体教室才有意义。因此它们是强整体-部分关系，用组合比普通关联更合理。

## 5.3 为什么 `Reservation` 同时关联用户、教室、时间段

预订记录本身就是一条完整业务结果，必须知道：

- 谁申请的
- 预订了哪个教室
- 占用了哪个时间段

因此 `Reservation` 必须与这三个对象发生关系。

## 5.4 为什么引入 `ClassroomManagementRecord` 和 `UserManagementRecord` 拆分多对多关系

在系统静态模型与数据库建模中，**多对多（N:M）关系无法直接在物理表或简洁的单向对象关系中优雅落地**。如果直接将 `Administrator` 与 `Classroom` / `User` 建立多对多关联：
- 无法在类中优雅地通过字段双向持有对方的无限制集合。
- 无法记录“哪个管理员在什么时间对哪间教室或哪个用户进行了什么具体操作”。

因此，本系统引入了两个中间日志实体类进行 **N:M 拆分解耦**：
1. **`ClassroomManagementRecord`（教室管理记录）**：将管理员与教室的多对多管理关系拆分为：
   - `Administrator` (1) --- (0..*) `ClassroomManagementRecord` （一对多）
   - `ClassroomManagementRecord` (0..*) --- (1) `Classroom` （多对一）
2. **`UserManagementRecord`（用户管理记录）**：将管理员与普通用户的多对多管理关系拆分为：
   - `Administrator` (1) --- (0..*) `UserManagementRecord` （一对多）
   - `UserManagementRecord` (0..*) --- (1) `User` （多对一）

这样设计带来的巨大优势包括：
- **可审计性（Audit Log）**：系统天然具备了管理员操作痕迹跟踪能力，完美记录了操作时间、操作类型（新增、注销、时段设置）和具体备注。
- **数据库结构规范化（NF）**：完美消除了多对多物理表，使其在实验3的数据库建模中直接落地为两个带外键的高内聚“操作关系表”，彻底避免了数据冗余和更新异常。

## 5.5 为什么增加服务类

如果把所有操作都塞进实体类里，会导致类过于臃肿。例如：

- 用户类既管账号又管预订流程
- 教室类既管教室状态又管复杂冲突校验
- 管理员类既表达角色又包办所有业务流程

这不符合高内聚、低耦合的原则。因此将认证、用户管理、教室管理、预订管理拆分到服务类，是更合理的面向对象设计方式。

## 6. 面向 Rational Rose 7.0 逆向生成的说明

为了便于你在 Rational Rose 7.0 中逆向生成类图，本次 Java 代码采用了以下策略：

- 使用英文类名，避免中文类名影响 Java 兼容性
- 使用较基础的 Java 写法，尽量兼容老版本工具解析
- 通过字段和方法参数表达类之间的关系
- 通过继承结构直接表达父子类关系

但要注意：**Java 代码逆向生成 UML 时，继承和普通关联最容易自动识别；组合关系、多重性、依赖方向有时仍需要你在 Rose 中手工微调。**

因此，在 Rose 逆向完成后，你需要重点检查并适当调整以下内容：

- `Classroom` 与 `ClassroomTimeSlot` 是否被标成组合
- 关联线的多重性是否正确显示为 `1`、`0..*`
- 服务类与实体类之间是否以依赖关系展示
- 图中类的摆放位置是否清晰

## 8. 最后总结

本次类图设计将教室预订系统划分为两大部分：

- **实体类层**：负责表示稳定的业务对象
- **服务类层**：负责表示系统中的管理流程与业务控制

这种设计方式既符合实验要求，也符合面向对象设计的一般原则，能够较好地支撑 Rational Rose 7.0 逆向生成类图和后续实验3的数据建模工作。

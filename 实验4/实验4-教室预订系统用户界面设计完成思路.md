# 实验4：教室预订系统用户界面设计完成思路

> 本文件是“实验4：用户界面设计（Axure RP）”的完成思路与操作指引。
> 它在实验1（需求/用例）、实验2（类图）、实验3（数据库）已完成成果的基础上编写，目的是让你在动手画原型前就清楚“画哪些页面、每个页面放什么、怎么用 Axure 做出来、最后交什么”，做到**前后一致、逐层细化**。

---

## 1. 实验定位与目标

### 1.1 指导书要求（实验4）

- 实验环境：Windows + **Axure RP** 快速原型设计工具。
- 实验目的：了解 Axure RP 的组成与功能；掌握 Axure RP 中 UI 设计工具的使用方法。
- 实验内容：完成**教室预订系统**的用户界面设计。
- 实验要求：设计并绘制上述教室预订系统的用户界面。

### 1.2 本实验的本质

实验4 **不是写可运行代码**，而是用 Axure RP 把系统的**主要页面 + 页面之间的跳转 + 关键交互**做成原型。评分关注点通常是：

1. **页面是否覆盖核心业务**（实验1的用例是否都有对应界面）。
2. **操作流程是否合理**（页面跳转符合真实使用顺序）。
3. **字段/命名是否与前面实验一致**（同一套用户、教室、时间段、预订记录）。
4. **角色差异是否体现**（管理员端 vs 师生端权限不同）。

> 一句话原则：**界面从用例来，字段从数据库来，权限从角色来。**

## 2. 设计依据（与实验1–3 的对应关系）

界面不要凭空想，必须复用前三个实验已经确定的元素。以下是从已完成成果中提炼出的“统一词表”，做原型时所有标签、字段名都从这里取。

### 2.1 角色（来自实验1 / 实验2）

- `管理员（Administrator）`：用户管理、教室管理、设置可用时间段、查询预订情况。
- `师生用户（TeacherStudentUser）`：登录、修改密码、查询空闲教室、预订教室。
- 抽象父类 `用户（User）`：承载登录、改密等公共行为。

### 2.2 核心业务对象与字段（来自实验2 类图 / 实验3 建表脚本）

界面上的输入框、列表列、详情字段，请直接对应下面的真实字段，保证“界面字段 = 数据库字段”。

| 业务对象 | 数据表 | 关键字段（界面会用到的） |
|---|---|---|
| 用户 | `t_user` | `user_id`、`account` 账号、`user_name` 姓名、`password` 密码、`contact_info` 联系方式、`user_status` 状态(ACTIVE/DISABLED) |
| 师生用户 | `t_teacher_student` | `ts_id`、`identity_type` 身份(TEACHER/STUDENT) |
| 教室 | `t_classroom` | `classroom_id`、`classroom_name` 名称、`location` 位置、`capacity` 容量、`equipment_info` 设备、`classroom_status` 状态(AVAILABLE/DISABLED) |
| 教室可用时间段 | `t_classroom_time_slot` | `slot_id`、`use_date` 日期、`start_section` 开始课时、`end_section` 结束课时、`slot_status` 状态(FREE/RESERVED/UNAVAILABLE) |
| 预订记录 | `t_reservation` | `reservation_id`、`ts_id` 申请人、`classroom_id` 教室、`slot_id` 时段、`purpose` 用途、`reservation_status` 状态(CREATED/CANCELLED/FINISHED)、`create_time` 创建时间 |

### 2.3 可直接落到页面的查询结构（来自实验3 视图/存储过程）

- 视图 `v_available_classroom`（可用教室 + 空闲时段）→ **空闲教室查询页**的结果表格。
- 视图 `v_reservation_detail`（预订 + 申请人 + 教室 + 时段）→ **预订情况查询页 / 我的预订页**的结果表格。
- 存储过程 `sp_reserve_classroom`（事务内校验并写入预订）→ **预订教室页**“提交预订”按钮背后的逻辑（原型里用交互模拟成功/冲突两种结果即可）。

> 关键业务约束（必须在界面上体现）：
> - 时间精确到“**课时**”：用 `开始课时 / 结束课时` 选择，而不是随意填时间。
> - “空闲”= 教室 `AVAILABLE` 且 时段 `FREE`：查询结果只列同时满足两者的教室。
> - 同一教室、同一日期、同一课时**不可重复预订**：提交时给出“时段冲突，预订失败”的异常提示。

## 3. 页面清单（Sitemap）

按角色组织。建议在 Axure 左侧 Pages 面板就按这个层级建好页面树。

```
教室预订系统
├─ 0. 登录页（Login）                         [公共]
├─ 1. 修改密码页（Change Password）            [公共，登录后]
│
├─ 师生端
│   ├─ S0. 师生首页 / 主菜单
│   ├─ S1. 空闲教室查询页（核心）
│   ├─ S2. 教室预订页（核心）
│   ├─ S3. 预订结果提示页 / 弹窗
│   └─ S4. 我的预订（查看/取消）
│
└─ 管理员端
    ├─ A0. 管理员首页 / 控制台
    ├─ A1. 用户管理页（增加/停用/查询用户）
    ├─ A2. 教室管理页（增加/删除/停用教室）
    ├─ A3. 教室可用时间段设置页
    └─ A4. 预订情况查询页
```

- **必做页面（建议优先完成，最容易拿分）**：登录页、空闲教室查询页、教室预订页、用户管理页、教室管理页、预订情况查询页。
- **加分页面**：修改密码页、两个角色首页、可用时间段设置页、我的预订页、结果提示弹窗。

## 4. 页面流程（导航关系）

### 4.1 师生端主流程

```
登录 → 师生首页 → 空闲教室查询(选日期+课时+条件→查询) 
     → 从结果列表点“预订” → 教室预订页(确认信息+填用途) 
     → 提交 → 成功提示 → 我的预订(可取消)
```

### 4.2 管理员端主流程

```
登录 → 管理员首页(控制台) → 
   ├─ 用户管理(增加/停用用户)
   ├─ 教室管理(增加/删除/停用教室)
   ├─ 可用时间段设置(选教室→设日期+课时段)
   └─ 预订情况查询(按日期/教室/状态筛选→看表格)
```

> 做原型时，**登录页要按角色分流**：管理员登录后进 A0，师生登录后进 S0。可以用两个测试账号或一个“角色”单选来模拟跳转。

## 5. 各页面详细设计

下面给出每个核心页面应包含的控件、字段（对齐第 2 节词表）和关键交互。

### 5.1 登录页（Login）

- 控件：账号输入框（`account`）、密码输入框（`password`，掩码）、登录按钮、（可选）角色选择。
- 交互：
  - 点击“登录” → 校验非空 → 按角色跳转（管理员→A0，师生→S0）。
  - 失败提示：账号不存在 / 密码错误 / 账号已停用（对应 `user_status=DISABLED`）。

### 5.2 修改密码页（Change Password）

- 控件：旧密码、新密码、确认新密码、保存按钮。
- 交互：旧密码错误、两次不一致、格式不合规的提示；保存成功提示。

### 5.3 师生首页（S0）

- 顶部：欢迎信息 + 当前用户姓名（`user_name`）。
- 菜单：空闲教室查询、我的预订、修改密码、退出。

### 5.4 空闲教室查询页（S1，核心）

- 查询条件区：
  - 日期选择（`use_date`）。
  - 课时范围：开始课时 `start_section`、结束课时 `end_section`（下拉，如 第1节~第12节）。
  - 可选筛选：容量 `capacity`、设备 `equipment_info`、位置 `location`。
  - “查询”按钮。
- 结果列表（表格/Repeater，对应视图 `v_available_classroom`）：
  - 列：教室名称、位置、容量、设备、可用时段（日期 + 起止课时）、操作（“预订”按钮）。
  - 空结果时显示“当前条件下无空闲教室”，不是报错。
- 交互：点“预订” → 携带该行 `classroom_id / slot_id` 跳转 S2。

### 5.5 教室预订页（S2，核心）

- 信息确认区（只读，来自上一页选择）：教室名称、位置、容量、日期、起止课时。
- 输入：用途 `purpose`（文本框）。
- 按钮：提交预订、返回。
- 交互（模拟 `sp_reserve_classroom`）：
  - 成功：生成预订记录（`reservation_status=CREATED`），跳 S3/弹窗“预订成功”。
  - 冲突：提示“该教室该课时已被预订，预订失败”（演示异常流）。

### 5.6 我的预订页（S4）

- 表格（对应 `v_reservation_detail`，按当前用户过滤）：预订编号、教室、日期、起止课时、用途、状态、申请时间、操作（取消）。
- 交互：点“取消”→ 二次确认 → 状态变为 `CANCELLED`。

### 5.7 管理员首页 / 控制台（A0）

- 菜单/卡片入口：用户管理、教室管理、可用时间段设置、预订情况查询、修改密码、退出。

### 5.8 用户管理页（A1）

- 工具区：新增用户按钮、按账号/姓名搜索。
- 列表：账号、姓名、身份(`identity_type`)、联系方式、状态(`user_status`)、操作（停用/启用）。
- 新增用户弹窗/表单：账号、姓名、初始密码、身份(教师/学生)、联系方式 → 保存（校验账号唯一）。
- 交互：停用用户前若有未完成预订，提示“该用户存在未完成预订，是否继续”。

### 5.9 教室管理页（A2）

- 工具区：新增教室按钮、按名称/位置搜索。
- 列表：教室编号、名称、位置、容量、设备、状态(`classroom_status`)、操作（停用/删除、设置时间段）。
- 新增/编辑教室表单：名称、位置、容量、设备信息 → 保存（编号唯一）。
- 交互：删除/停用前检查是否存在未来有效预订；推荐用“停用(逻辑删除)”表达。

### 5.10 教室可用时间段设置页（A3）

- 选择教室（`classroom_id`）。
- 设置区：日期 `use_date`、开始课时 `start_section`、结束课时 `end_section` → “添加时段”。
- 已设置时段列表：日期、起止课时、状态(`slot_status`)、操作（删除/置为不可用）。
- 交互：新增时段与已有时段冲突时给出提示；已被预订(`RESERVED`)的时段不可直接改为不可用。

### 5.11 预订情况查询页（A4）

- 筛选条件：日期范围、教室、预订状态(`reservation_status`)。
- 结果表格（对应 `v_reservation_detail`）：预订编号、申请人姓名、账号、身份、教室、位置、日期、起止课时、用途、状态、创建时间。
- 交互：查询、导出（可选）。

## 6. Axure RP 操作要点（如何把上面的设计做出来）

> 下面是把“设计”落成“原型”的具体做法，按这个顺序操作效率最高。

1. **建页面树**：在左侧 `Pages` 面板，按第 3 节 Sitemap 把所有页面建好并命名。

2. **做公共母版（Masters）**：
   - 顶部栏（系统名 + 当前用户 + 退出）做成一个 Master。
   - 左侧/顶部导航菜单做成 Master，师生端、管理员端各一套，拖到对应页面即可统一风格。

3. **表格用中继器（Repeater）**：
   - 空闲教室、预订记录、用户列表、教室列表都用 `Repeater` 控件，定义数据集列（对齐第 2 节字段），这样行可复制、列对齐、最像真实系统。
   - 简化做法（够用）：直接用 `Table` 控件手填几行示例数据演示效果。

4. **查询/提交交互（Interactions/Cases）**：
   - 给“查询/登录/提交/保存”按钮加 `Click → Case`：用条件 `Condition` 判断（如必填非空、角色判断），再 `Open Link / Show 弹窗 / Set Text` 给出结果。
   - 登录按角色跳转：用 Condition 判断输入或单选，分别 `Open Link` 到 A0 或 S0。

5. **弹窗/提示用动态面板（Dynamic Panel）**：
   - “预订成功 / 冲突失败 / 删除确认”等做成隐藏的 Dynamic Panel，点击时 `Show`，体现成功流与异常流两种结果。

6. **状态体现**：列表里的 `状态` 列直接用文字（可用/停用、空闲/已预订、CREATED/CANCELLED）并配不同底色，体现实验1的数据一致性要求。

7. **风格设置（Style）**：统一字体（如指导书示例用“仿宋”）、表头底色、按钮位置固定（查询/提交/保存放右上或表单底部）。

8. **导出成果**：`Publish → Generate HTML files`（本地 HTML 原型）或导出各页面 PNG 图片，用于写报告和提交。

## 7. 设计规范与易错点

**规范建议：**
- 页面结构清晰，控件别堆太乱；同类页面（列表页、表单页）保持统一布局。
- 重要操作按钮位置固定：查询、提交、保存。
- 输入项命名与前面实验一致（账号/姓名/教室名称/课时/用途…）。
- 跳转逻辑符合真实使用顺序（先查询再预订，先登录再操作）。

**常见扣分点（务必避免）：**
- 页面不全：缺管理员端的用户/教室/时间段管理页。
- 时间不到“课时”：用了模糊的时间段而非“开始课时/结束课时”。
- 没体现角色差异：管理员和师生看到同一套界面。
- 字段和实验2/3 对不上：界面字段名与数据库字段名脱节。
- 只画静态图、没有任何跳转/提示交互。

## 8. 推进步骤（建议顺序）

1. 熟悉 Axure：拖几个控件、试试 Style / Properties / Interactions 面板。
2. 建好 Pages 树 + 公共母版（顶部栏、导航）。
3. 先做**登录页**并实现按角色跳转。
4. 做师生端主线：空闲教室查询 → 教室预订 → 成功/冲突提示。
5. 做管理员端主线：控制台 → 用户管理 → 教室管理 → 时间段设置 → 预订查询。
6. 补充：修改密码、我的预订、各类提示弹窗。
7. 统一风格 + 检查页面跳转闭环。
8. 导出 HTML / 截图，写报告（页面截图 + 跳转说明 + 关键页面设计说明）。

## 10. 一句话总结

实验4 就是把前三个实验“落地成看得见的界面”：**用例决定有哪些页面，数据库字段决定页面上放什么，角色决定谁能看到什么**。只要页面覆盖完整、流程闭环、字段命名和前面一致、并体现“课时粒度 + 空闲判定 + 冲突校验 + 角色差异”这几条业务规则，实验4 就既完整又成体系。
